Legibility: interaction in AI interfaces

As software moves from deterministic execution to probabilistic processes, motion becomes the primary mechanism for making work visible.

Introduction

Most interface patterns assume a particular kind of system.

One where operations are fast, bounded, and deterministic. Work begins, progresses, and completes in a predictable way. Under these conditions, interfaces can afford to abstract the process. A small set of signals — loading, success, failure — is sufficient to describe what is happening.

The user does not need to understand the work. Only that it will complete.

AI systems do not conform to these assumptions.

Their latency is variable, their execution is non-deterministic, and their behavior unfolds over time. A response may involve multiple stages — retrieval, generation, tool use — whose duration and outcome are not known in advance. The system is not simply “working” or “done.” It is progressing, revising, and sometimes failing in ways that are observable if surfaced.

Under these conditions, abstraction breaks down.

Interfaces can no longer summarize system behavior into a single state without losing essential information. Users need to understand not just that work is happening, but what kind of work is happening — whether a system is actively progressing, stalled, or producing partial results.

This creates a new requirement: interfaces must represent ongoing computation, not just signal its existence.

The spinner is a canonical example of the previous model. It indicates that a process has started without describing the process itself. For systems where execution is short and predictable, this is sufficient.

For AI systems, it is not.

The spinner

A spinner communicates liveness without structure. It indicates that a process has started, but provides no information about its state, progress, or expected duration.

For short operations, this is acceptable. The interaction completes before the absence of information becomes meaningful.

As latency increases, the limitation becomes structural. The spinner does not distinguish between active computation, retry loops, or failure states. It continues to signal progress even when none is occurring.

The issue is not visual design. The issue is the model of work it represents: fast, linear, and opaque.

The typing indicator

The first adaptation was to borrow from human messaging: the typing indicator.

This pattern implies continuous, real-time composition. In messaging systems, that implication is accurate. In AI systems, it is often not.

In many implementations, the indicator runs during network latency or preprocessing, while the response is generated off-screen and delivered in full. The motion implies continuity; the system behaves discretely.

Users detect this mismatch quickly. When the entire response appears at once, the indicator is revealed as performance rather than representation.

Patterns that misrepresent the underlying process do not survive.

1. Reveal-as-artifact

The output is the progress.

There is no loading state. The artifact — text, image, or code — appears in incomplete form and resolves over time.

In language models, this takes the form of token streaming. Text is generated sequentially, with variable latency between tokens. The interface exposes that process directly.

Token streaming
Idle
Explain why streaming tokens helps users trust AI.

This representation is effective because it is faithful. Users observe pacing, hesitation, and acceleration as they occur.

The same pattern appears across modalities:

  • Image generation through progressive denoising
  • Video generation through incremental rendering
  • Code generation through inline ghost text

In each case, progress is identical to the artifact under construction.

The pattern fails when this relationship is broken. Buffered output — released all at once after a delay — simulates progress without representing it.

2. Progress-as-steps

Work is staged into discrete phases.

When processes have identifiable structure, interfaces expose it:

  • Queue → sample → upscale
  • Shot-by-shot video rendering
  • File-by-file code edits

This allows users to localize delay. It answers not just whether the system is slow, but where it is slow.

The pattern depends on structural accuracy. If the stages shown do not correspond to actual execution, the interface becomes misleading.

Linear progress bars are a common failure. They imply steady advancement even when the underlying process is uneven or stalled.

In these cases, the interface does not represent progress — it performs it.

3. Show-the-work

Intermediate computation is exposed.

Rather than summarizing a process, the interface reveals its internal steps:

  • Tool-call logs
  • Retrieval steps
  • Prompt transformations
  • Reasoning traces

This pattern addresses a different question than progress indicators. It does not answer “how far along is the system?” but “what is the system doing?”

Its primary function is inspectability. Users can compare intermediate steps against final outputs and identify inconsistencies.

In practice, this information is often collapsed by default. What matters is not constant visibility, but availability.

The pattern fails when exposure is selective. If only coherent or successful steps are shown, the interface presents a reconstructed narrative rather than the actual process.

4. Ambient presence

Only liveness is communicated.

When no meaningful intermediate state can be exposed, interfaces fall back to minimal signals:

  • Pulses
  • Shimmers
  • Breathing indicators

These communicate that the system is active without making claims about what it is doing.

This is appropriate when no structured progress exists.

It fails when used for extended durations. At that point, ambient motion collapses back into the spinner’s limitation: presence without explanation.

Agents

These patterns become more consequential in agent systems.

A chat interface produces an answer. An agent executes actions.

The difference is operational. Errors in chat systems produce incorrect outputs. Errors in agent systems produce external effects: sent messages, modified files, executed transactions.

This raises the requirement for legibility.

Agent interfaces must represent three distinct temporal views of the same process:

  • Plan — what the system intends to do
  • Trace — what the system is doing
  • Receipt — what the system has done

These correspond to three fundamental functions: planning, monitoring, and auditing.

The plan

A structured description of intended actions presented before execution.

Users can approve, modify, or reject it. Once execution begins, the plan becomes a reference against which progress is measured.

Observability

These patterns are not new. They originate in observability systems.

Observability tools were designed to make distributed, unreliable processes understandable in real time. They rely on a set of primitives:

  • Metrics
  • Logs
  • Traces
  • Topology
  • Experiments

Each represents system behavior over time.

Live metrics make trends visible. Log streams expose event sequences. Traces reveal structured execution paths. Topology maps show relationships between components.

The shared function is to make ongoing computation legible without reducing it to a single state.

AI interfaces are converging on this model because they face the same constraint. They are interfaces to systems whose behavior cannot be summarized at a glance.

In this sense, AI interfaces are becoming consumer-facing observability tools.

The observability anti-pattern

Aggregate numbers without motion.

A single value — average latency, success rate — collapses temporal variation into a static summary. It obscures spikes, regressions, and failure modes.

Without motion, there is no context. Without context, system behavior cannot be interpreted.

If an interface would appear unchanged over time, it is not observing. It is reporting.

Misrepresentations of uncertainty

Two failure modes recur across these patterns.

Confidence performance

Interfaces that present probabilistic outputs with deterministic affordances — definitive typography, success states — miscalibrate user trust.

The system’s uncertainty is replaced by the interface’s certainty.

Premature resolution

Interfaces that signal completion for processes that remain revisable discourage verification.

A filled progress bar or a final-looking answer suggests that work is finished when it is not.

Conclusion

The evolution of motion in AI interfaces reflects a mismatch between interface conventions and system properties.

Traditional patterns — spinners, progress indicators, completion states — assume systems that are fast, predictable, and opaque. They reduce work to a binary state.

AI systems do not conform to these assumptions. Their behavior is variable, their execution is multi-stage, and their outputs are often provisional.

This shift aligns AI interfaces more closely with observability systems than with traditional application design. In both cases, the objective is to make processes interpretable over time.

The shift is not that interfaces now look different. It is that interfaces now have to carry information they previously hid. Motion is the channel through which that information travels.
